  1. React的严格模式如何使用，有什么用处
    StrictMode 是一个用来突出显示应用程序中潜在问题的工具,不会渲染任何可见的 UI,

    • 识别不安全的生命周期
    • 关于使用过时字符串 ref API 的警告
    • 关于使用废弃的 findDOMNode 方法的警告
    • 检测意外的副作用
    • 检测过时的 context API

  2. 为什么使用jsx的组件中没有看到使用react却需要引入react

    • 本质上来说JSX是React.createElement(component, props, …children)方法的语法糖, React 17之后，就不再需要引入，因为 babel 已经可以帮我们自动引入react

  3. React.Children.map和js的map有什么区别

    • JavaScript中的map不会对为null或者undefined的数据进行处理，而React.Children.map中的map可以处理React.Children为null或者undefined的情况

  4. React 中的高阶组件运用了什么设计模式

    • 装饰模式： 装饰模式的特点是不需要改变 被装饰对象 本身，而只是在外面套一个外壳接口（类似添加一些属性，方法）

  5. 类组件和函数组件有何不同

    • 类组件有生命周期，有状态
    • 函数组件是无状态的，它们渲染 UI 的首选只依赖于属性，因为它们比基于类的组件更简单、更具性能

  6. React 中 keys 的作用是什么

    • 用于追踪哪些元素被修改，被添加或者被移除的辅助标识，最好使用一个唯一标识，不容易变化的标识

  7. 为什么调用 setState 而不是直接改变 state

    • 因为react中数据的不变性，底层是用diff实现，数据不变性被破坏，会出现各种难以注意到的问题
    • 方便做性能优化，自己控制UI需不需要更新

  8. React的事件和普通的HTML事件有什么不同

    • 对于事件名称命名方式，原生事件为全小写，react 事件采用小驼峰
    • 对于事件函数处理语法，原生事件为字符串，react 事件为函数
    • react 事件不能采用 return false 的方式来阻止浏览器的默认行为，而必须要地明确地调用preventDefault()来阻止默认行为

    合成事件是 react 模拟原生 DOM 事件所有能力的一个事件对象，所有的合成时间都是绑定到document上面，然后用dispatchEvent分发

    • 兼容所有浏览器，更好的跨平台
    • 将事件统一存放在一个数组，避免频繁的新增与删除（垃圾回收）
    • 方便 react 统一管理和事务机制

  9. React 组件中怎么做事件代理？它的原理是什么

    • react基于VDom实现一个合成事件层，定义的事件处理器会接收到一个合成事件对象的实例，与原生对象具有同样的接口，支持冒泡机制，所有的事件都自动绑定在最外层上。
    • 事件机制分为两个阶段，事件注册和事件分发，所有的事件都会注册到document上，然后使用统一的回调函数dispatchEvent分发

  10. React 高阶组件、Render props、hooks 有什么区别，为什么要不断迭代

    • 高阶组件是react中用于复用组件逻辑的一种高级技巧，参数为组件，返回新组件的函数
    • props是告知组件使用什么内容渲染组件。
    • hooks与高阶组件和props类似，只是更加的简洁，并且能帮助较少嵌套。

  11. 对React-Fiber的理解，它解决了什么问题
    React V15 在渲染时，会递归比对 VirtualDOM 树，找出需要变动的节点，然后同步更新它们， 一气呵成。这个过程期间， React 会占据浏览器资源，这会导致用户触发的事件得不到响应，并且会导致掉帧，导致用户感觉到卡顿。
    为了给用户制造一种应用很快的“假象”，不能让一个任务长期霸占着资源。 可以将浏览器的渲染、布局、绘制、资源加载(例如 HTML 解析)、事件响应、脚本执行视作操作系统的“进程”，需要通过某些调度策略合理地分配 CPU 资源，从而提高浏览器的用户响应速率, 同时兼顾任务执行效率。
    所以 React 通过Fiber 架构，让这个执行过程变成可被中断。“适时”地让出 CPU 执行权，除了可以让浏览器及时地响应用户的交互，还有其他好处

    • 分批延时对DOM进行操作，避免一次性操作大量 DOM 节点，可以得到更好的用户体验
    • 给浏览器一点喘息的机会，它会对代码进行编译优化（JIT）及进行热代码优化，或者对 reflow 进行修正
      核心思想： Fiber也称协程或者纤程，他和纤城不一样，协程本身是没有并发或者并行能力的（需要配合线程），他只是一种控制流程的让出机制，让出CPU执行权，让cpu能在这短时间执行其他的操作，渲染的过程可以被中断，可以将控制权交给浏览器，让位给高级优先级的任务，浏览器空闲后在回复渲染。

  12. Component, Element, Instance 之间有什么区别和联系

    • element： 元素，React元素是一个普通对象，描述了一个DOM节点或者其他Component，元素创建后是不可变的。
    • Component: 组件，一个react组件可以有多种方式声明，可以是带有render的类，也可以是简单的函数，返回元素树
    • Instance: 实例，一个实例就是Component class中是使用的关键字this，它用来存储状态和生命周期事件
    • 函数组件中没有Instance实例，类组件中有Instance

  13. React.createClass和extends Component的区别有哪些

    • 语法区别：

      • createClass本质上一个工厂函数，extends更类似与ES6中的class写法，语法差别在方法的定义和静态属性声明上（createClass是用逗号隔开）

    • propType 和 getDefaultProps：

      • React.createClass：通过proTypes对象和getDefaultProps()方法来设置和获取props.
      • React.Component：通过设置两个属性propTypes和defaultProps

    • 状态的区别：

      • React.createClass：通过getInitialState()方法返回一个包含初始值的对象
      • React.Component：通过constructor设置初始状态

    • this的区别

      • React.createClass：会正确绑定this
      • React.Component：由于使用了 ES6，这里会有些微不同，属性并不会自动绑定到 React 类的实例上，需要自己指定this

  14. 对React的插槽(Portals)的理解，如何使用，有哪些使用场景

    Portal 提供了一种将子节点渲染到存在于父组件以外的 DOM 节点的优秀的方案=

    • 可以用在对话框以及提示框，或者模态框等等

  15. Redux 原理及工作流程

    • Redux设计思想：利用可预测状态控制UI
    • Redux 是 一个统一的状态管理容器，提供可预测化的状态管理，由compose.js 提供从右到左进行函数式编程，createStore.js 提供作为生成唯一store的函数，combineReducers.js 提供合并多个reducer的函数，保证store的唯一性，bindActionCreators.js 可以让开发者在不直接接触dispacth的前提下进行更改state的操作，applyMiddleware.js 这个方法通过中间件来增强dispatch的功能
    • 数据是单向流动，防止变化
    • 工作流程：action进行dispatch -> store -> reducer处理action，返回新的state

  16. Redux 状态管理器和变量挂载到 window 中有什么区别

    两者都是存储数据以供后期使用。但是Redux状态更改可回溯——Time travel，数据多了的时候可以很清晰的知道改动在哪里发生，完整的提供了一套状态管理模式。而window的变量管理比较混乱,维护不当还会造成变量污染,不适合中大型项目的开发.

  17. React的生命周期

    defaultProps -> constructor -> componentWillMount -> render -> componentDidMount -> componentWillReceiveProps -> shouldComponentUpdate -> componentWillUpdate -> componentDidUpdate -> componentWillUnmount

  18. componentWillReceiveProps的出触发条件

    • 当component接收到新的props的时候调用

  19. React16.3对生命周期的改变
    react 16.3+废弃了 componentWillMount, componentWillReceiveProps, componentWillUpdate,
    取而代之的两个新的生命周期，getDerivedStateFromProps， getSnapshotBeforeUpdate

  20. 接入redux的过程

    • 利用createStore创建一个总状态池
    • 利用applyMiddleware和redux-thunk创建一个可处理复杂antion的中间件
    • 利用react-redux把状态池store绑定到根节点
    • 页面根据react-redux的connect把状态绑定到props

  21. 绑定connect的过程

    • 类似高阶组件，就是传入一个component，connect对component在不改变原本属性，状态的前提下，绑定一些新的props
